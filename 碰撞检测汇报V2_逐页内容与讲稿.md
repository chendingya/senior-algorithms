## 第4页 | 算法一：四叉树 —— 设计思想

### 画面显示

- 标题：**四叉树（Quadtree）—— 分治空间，化整为零**
- 设计策略标签：**分治法（Divide and Conquer）**
- 左侧：四叉树分裂动画/示意图
  - 第一步：整个2D地图
  - 第二步：对象数超阈值 → 分裂为4个象限
  - 第三步：递归分裂，直到每个叶节点对象数 ≤ 阈值
- 右侧：核心思想
  - "让每个对象只和局部邻居打交道"
  - 与直觉的关系：类似"把地图分区，每个区域配一个管理员"
  - 对象少的区域不浪费计算，对象密的区域自动细分
- 底部表格：

| 核心动作 | 设计策略 | 适用场景 | 不适用场景 |
|---------|---------|---------|-----------|
| 递归分裂空间 | 分治 | 2D偏静态、分布稀疏 | 高动态、对象密集 |

### 讲稿（~45秒）

> 四叉树的设计思想是"分治"——把二维地图递归划分为四个象限，区域内对象超阈值时继续分裂。直觉是：稀疏区域的对象不必和其他区域比较。查询时只在目标叶节点及相邻区域搜索，大幅缩小范围。这类似给地图分区管理——每个管理员只负责自己地盘。四叉树非常适合二维稀疏场景。

## 第5页 | 四叉树 —— 伪代码与复杂度分析

### 画面显示

- 标题：**四叉树 —— 算法流程与复杂度**
- 左侧伪代码：

```
// 插入操作
func Insert(node, obj):
    if node.count < MAX_CAPACITY:
        node.objects.add(obj)
        return
    if not node.isDivided:
        node.Subdivide()          // 分裂为4个子节点
    for child in node.children:
        if child.boundary.contains(obj):
            Insert(child, obj)
            return

// 范围查询操作
func Query(node, range, results):
    if not node.intersects(range):
        return                     // 剪枝
    for obj in node.objects:
        if range.contains(obj):
            results.add(obj)
    if node.isDivided:
        for child in node.children:
            Query(child, range, results)
```

- 右侧复杂度分析表：

| 操作 | 最好情况 | 平均情况 | 最坏情况 | 空间 |
|-----|---------|---------|---------|-----|
| 构建（n次插入） | O(n log n) | O(n log n) | O(n²) | O(n) |
| 单次查询 | O(log n) | O(log n) | O(n) | — |
| 全场景碰撞检测 | O(n log n) | O(n log n) | O(n²) | O(n) |

- 最坏情况说明（小字）：所有对象重叠在同一区域 → 树退化为链表 → 退化为暴力法
- 关键依赖：对象分布是否均匀，阈值 MAX_CAPACITY 的选取

### 讲稿（~45秒）

> 插入时，对象数未超限就直接存入，超限则分裂后递归插入。查询时，节点与范围不相交则直接剪枝，否则搜索当前对象和子节点。平均情况下，单次查询O(log n)，全场景检测O(n log n)。最坏情况——所有对象集中在同一区域——树会退化，退回到O(n²)。所以四叉树的效果高度依赖空间分布是否均匀。

## 第6页 | 四叉树 —— 正确性与取舍

### 画面显示

- 标题：**四叉树 —— 正确性与优劣取舍**
- 正确性说明（左侧）：
  - **不变量**：每个对象一定存储在某个叶节点（或未分裂节点）中
  - **空间完备性**：四个子节点的区域之和 = 父节点区域，无遗漏
  - **查询正确性**：如果对象在查询范围内，它所在的叶节点一定与查询范围相交 → 一定会被找到
- 优劣对比（右侧）：

| 优势 | 劣势 |
|------|------|
| 结构直观，适合二维空间查询 | 对象频繁移动时，树需要反复删除/插入/分裂 |
| 稀疏场景下显著减少比较次数 | 极端分布下树深膨胀，维护成本吞掉收益 |
| 实现简单，易于理解 | 仅适用于2D（3D需要八叉树，结构更复杂） |

- 底部一句：**静态地图、动态对象少 → 四叉树是好选择；高动态场景 → 需要其他方案**

### 讲稿（~40秒）

> 四叉树的正确性基于一个不变量：每个对象一定被存储在某个节点中，且四个子节点的空间覆盖了父节点全部区域，不会遗漏。查询时，只要对象在范围内，所在叶节点一定与查询范围相交，保证能被找到。但四叉树有两个核心问题：高动态场景中对象频繁移动，需要不断删除和重新插入，维护代价很高；对象分布极不均匀时，树深急剧增加，丧失加速效果。因此四叉树更适合偏静态的二维场景。

## 第7页 | 算法二：SAP —— 设计思想

### 画面显示

- 标题：**Sweep and Prune（SAP）—— 降维打击，排序复用**
- 设计策略标签：**降维 + 利用时间局部性（Temporal Coherence）**
- 核心思想图解（流程图）：

```
第1步：取每个对象的 AABB，投影到 X 轴 → 得到 N 个区间
第2步：对区间按左端点排序
第3步：从左往右扫描，重叠的区间 = 候选碰撞对

  对象A  |████████|
  对象B      |████████|      ← A和B在X轴重叠 → 候选对
  对象C                          |████████|  ← C和A/B不重叠 → 排除
```

- 关键洞察（醒目文字）：
  - **"帧与帧之间，对象位置变化很小 → 排序顺序几乎不变 → 插入排序接近O(n)"**
  - 这就是时间局部性（Temporal Coherence）
  - 历史：又称"Sort and Sweep"（Baraff, 1992博士论文）；I-COLLIDE（Cohen et al., 1995）首次系统化
- 核心优化原理——**不等式传递性**：

```
若已知 A.right ≤ B.left 且 B.right ≤ C.left
→ 由传递性：A.right ≤ C.left
→ 无需检测 A 与 C，直接跳过！
→ 排序让这个剪枝可以大规模生效
```

- 底部表格：

| 核心动作 | 设计策略 | 适用场景 | 关键优势 |
|---------|---------|---------|---------|
| 投影+排序+扫描 | 降维+时间局部性 | 高动态、大量对象 | 排序结果可跨帧复用 |

### 讲稿（~45秒）

> SAP的设计思想是"降维"——把二维碰撞简化为一维区间重叠问题。将物体包围盒投影到坐标轴，排序后扫描找重叠。这个算法最早出自Baraff 1992年博士论文，后由Cohen等人1995年I-COLLIDE系统化。关键洞察有两个：一是利用时间局部性——帧间变化小，插入排序接近O(n)；二是利用不等式传递性——A与B不重叠、B与C不重叠，则A与C一定不重叠，可直接跳过，实现大规模剪枝。

## 第8页 | SAP —— 伪代码与复杂度分析

### 画面显示

- 标题：**SAP —— 算法流程与复杂度**
- 左侧伪代码：

```
func SweepAndPrune(objects):
    // 1. 沿X轴按AABB最小X坐标排序
    Sort(objects, key = aabb.minX)  // 首次：快排；后续帧：插入排序

    candidates = []
    // 2. 从左到右扫描
    for i = 0 to n-1:
        for j = i+1 to n-1:
            // 3. 剪枝：后续对象的minX > 当前maxX → 后面都不重叠
            if objects[j].minX > objects[i].maxX:
                break
            // 4. X轴重叠 → 检查Y轴（和Z轴）
            if AABBOverlap(objects[i], objects[j]):
                candidates.add(objects[i], objects[j])
    return candidates
```

- 右侧复杂度分析：

| 操作 | 最好情况 | 平均情况 | 最坏情况 | 空间 |
|-----|---------|---------|---------|-----|
| 首次排序 | O(n log n) | O(n log n) | O(n log n) | O(n) |
| 后续帧排序（插入排序） | **O(n)** | O(n) | O(n log n) | O(n) |
| 扫描生成候选对 | O(n) | O(n + m) | O(n²) | O(m) |
| **全场景碰撞检测（首次）** | O(n log n) | **O(n log n + m)** | O(n²) | O(n+m) |
| **全场景碰撞检测（后续帧）** | **O(n)** | **O(n + m)** | **O(n²)** | O(n+m) |

- 注：m = 某轴上的重叠区间对数（远小于n²，典型游戏中 m ≪ n²）
- 关键：复杂度部分**与输出规模挂钩**（output-sensitive），碰撞对越少越快
- 最好情况：对象几乎不动，排序顺序不变，扫描仅线性 → **O(n)**
- 最坏情况：对象剧烈穿插导致完全乱序，排序退化为O(n log n)，且区间大量重叠 → **O(n²)**
- 参考：Lean Rada 对SAP复杂度的详细分析（见参考资料）

### 讲稿（~45秒）

> SAP分两步：先排序，再扫描。首次帧用快速排序O(n log n)，后续帧序列几乎有序，改用插入排序接近O(n)。扫描时从左到右遍历，利用不等式传递性——若A与B不重叠、B与C不重叠，则A与C一定也不重叠——实现提前break终止内层循环，复杂度O(n + m)，m为实际重叠数。这和暴力法本质不同：SAP复杂度与输出规模挂钩（output-sensitive），碰撞对越少越接近O(n)。最坏情况下对象剧烈穿插完全乱序，才退化为O(n²)。

## 第9页 | SAP —— 正确性与适用讨论

### 画面显示

- 标题：**SAP —— 正确性分析与适用边界**
- 正确性说明：
  - **投影保交（分离轴定理）**：若A和B在2D/3D空间中相交，则它们在任意坐标轴上的投影必有重叠
  - **逆否命题**：若某轴上投影不重叠 → 一定不相交 → 可安全排除
  - **多轴验证**：实际实现中通常检查3个轴（X/Y/Z），三轴都重叠才是真正的候选对
  - **排序剪枝的正确性**：基于不等式传递性（a ≤ b 且 b ≤ c → a ≤ c），已排序列表上可安全跳过不可重叠的对
- 适用讨论：

| 适合SAP的场景 | 不适合SAP的场景 |
|-------------|---------------|
| 对象数量大但碰撞对少 | 对象密集且大量重叠（p ≈ n²） |
| 帧间运动幅度小 | 对象瞬间大幅位移（如传送） |
| 对象尺寸相近 | 对象尺寸差异极大（大区间吞没小区间） |

- 底部：**SAP的威力在于"变化少"；变化越大，越接近暴力法**

### 讲稿（~40秒）

> SAP的正确性基于分离轴定理：两物体空间相交则任意轴投影必有重叠；逆否命题——轴上不重叠则一定不相交——保证筛选安全。SAP的局限：对象密集大量重叠时候选对接近N²，扫描失效；对象尺寸差异大时大区间吞没小区间，同样退化。SAP最适合"大部分对象相安无事"的典型游戏场景。

## 第10页 | 算法三：GJK —— 设计思想（Minkowski差）

### 画面显示

- 标题：**GJK算法 —— 两体相交，以差判之，决于原点**
- 设计策略标签：**迭代逼近法（Iterative Approximation）**
- Minkowski差定义（大号公式）：

$$A \ominus B = \{a - b \mid a \in A, b \in B\}$$

- 关键等价关系（分三步推导，逐步显示）：

```
A ∩ B ≠ ∅ 
  ⟺ 存在公共点 p ∈ A 且 p ∈ B
  ⟺ p - p = 0 ∈ A ⊖ B
  ⟺ 原点 ∈ A ⊖ B

∴ 判断A和B是否相交 ⟺ 判断Minkowski差是否包含原点
```

- 图示：两个相交的凸形 → 它们的Minkowski差 → 原点在Minkowski差内部
- 底部：**核心转化：把"两个物体是否相交"转化为"一个形状是否包含原点"**

### 讲稿（~45秒）

> GJK的核心数学基础是Minkowski差：把A中每个点减去B中每个点，形成一个新的点集。关键性质：如果A和B有公共点p，则p-p=0，原点一定在Minkowski差中。反过来，若原点在Minkowski差中，说明存在a属于A、b属于B使得a=b，A和B相交。这样"两物体是否相交"的问题，就转化为"Minkowski差是否包含原点"。GJK不显式构造完整的Minkowski差——那太昂贵——而是通过迭代逼近在Minkowski差的边界上找关键点。

## 第11页 | GJK —— 算法流程与伪代码

### 画面显示

- 标题：**GJK —— 取支、立形、决原**
- 核心概念（左侧图解）：
  - **支持点（Support Point）**：沿给定方向在Minkowski差上最远的点
  - Support(A, B, d) = A上沿d最远的点 − B上沿−d最远的点
  - **单纯形（Simplex）**：线段(1D)、三角形(2D)、四面体(3D)
  - 算法不断更新单纯形，尝试包围原点
- 三步流程图：

```
初始化：找一个支持点 → simplex = [p₀]
循环：
  ① 找新支持点 p = Support(A, B, direction)
  ② 若 p·direction ≤ 0 → 原点不可能在Minkowski差中 → 返回 false
  ③ 将 p 加入单纯形
  ④ 更新单纯形和搜索方向：
     - 若单纯形已包含原点 → 返回 true
     - 否则 → 缩小搜索方向，继续迭代
```

- 右侧伪代码：

```
func GJK(shapeA, shapeB):
    dir = initialDirection()
    simplex = [Support(shapeA, shapeB, dir)]
    dir = -simplex[0]            // 方向指向原点

    while true:
        p = Support(shapeA, shapeB, dir)
        if dot(p, dir) <= 0:     // 新点无法超过原点
            return false         // 不相交
        simplex.append(p)
        if UpdateSimplex(simplex, ref dir):
            return true          // 单纯形包含原点 → 相交

func Support(shapeA, shapeB, dir):
    return shapeA.farthest(dir) - shapeB.farthest(-dir)
```

### 讲稿（~45秒）

> GJK的每一步只做一件事：沿当前搜索方向，在Minkowski差上找一个支持点。如果支持点没有越过原点，说明Minkowski差不含原点，直接判定不相交。越过了就加入单纯形，检查是否包围原点。2D场景单纯形依次是点、线段、三角形——三角形包围原点即判定相交。GJK的精妙之处在于不需构造完整Minkowski差，每次只需对两个凸体各做一次极值查询。

## 第12页 | GJK —— 复杂度分析

### 画面显示

- 标题：**GJK —— 复杂度分析**
- 核心结论（大字）：

```
GJK 的迭代次数有常数上界：2D ≤ 3次，3D ≤ 4次
→ 每对物体的判定时间取决于 Support 函数的代价
```

- 复杂度表：

| 因素 | 代价 | 说明 |
|------|-----|------|
| 迭代次数（2D） | O(1)，最多3次 | 单纯形最多3个点（三角形） |
| 迭代次数（3D） | O(1)，最多4次 | 单纯形最多4个点（四面体） |
| Support函数（朴素） | O(n)，n=顶点数 | 遍历所有顶点找极值 |
| Support函数（预处理） | O(log n) | 凸体预排序后二分查找 |
| **单次判定总代价** | **O(n)** 或 **O(log n)** | 取决于Support实现 |
| **增量模式（时间局部性）** | **接近O(1)** | 上一帧的单纯形作为起点，通常1-2次迭代收敛 |

- 空间复杂度：O(1)（仅存储当前单纯形，最多4个点）
- 关键文献发现（Montanari et al., 2017）：
  - GJK使用**Johnson距离子算法**计算单纯形中距原点最近的点，但存在数值稳定性问题（小量相乘导致精度损失）
  - Montanari, Petrinic & Barbieri 提出基于**有符号体积**的新子算法，避免小量相乘，CPU时间平均减少10%（接触场景改善5%-25%）

### 讲稿（~40秒）

> GJK的复杂度有一个关键事实：2D单纯形最多三角形（3个点），迭代≤3次；3D最多四面体，≤4次。迭代次数是常数，真正代价在Support函数。朴素实现需遍历所有顶点O(n)，预处理可降到O(log n)。更关键的是增量模式：物理引擎用上一帧单纯形作初始猜测，帧间变化小时通常1-2次迭代就收敛——使GJK在实际游戏中接近O(1)。空间复杂度恒为O(1)。正是这种稳定性和小开销使GJK成为实时碰撞检测的首选。

## 第13页 | GJK —— 正确性说明

### 画面显示

- 标题：**GJK —— 为什么这个算法是正确的？**
- 正确性三要素（逐步展示）：

**1. 等价转化（第10页已证）**
- A ∩ B ≠ ∅ ⟺ 原点 ∈ A ⊖ B

**2. 单纯形逼近的收敛性**
- Support函数保证每次取的点在Minkowski差的边界上
- **Johnson距离子算法**计算当前单纯形中距原点最近的点，保证每次迭代单纯形到原点的距离单调不增
- 有限维空间 + 距离单调不增 + 常数上界 → 必然收敛
- 注：Johnson子算法存在数值稳定性问题，2017年Montanari等人提出基于有符号体积的改进方案

**3. 判定的完备性**
- 返回 true：单纯形包含原点 → 原点确实在Minkowski差内 → 相交 ✓
- 返回 false：新支持点无法超越原点 → 整个Minkowski差在原点的一侧 → 不相交 ✓

- 底部：**不需要穷举所有顶点——每次只取"最有利于逼近原点"的点，这就是迭代贪心的力量**

### 讲稿（~40秒）

> GJK的正确性需要三件事。第一，等价转化已证明"相交"等价于"原点在Minkowski差中"。第二，Support函数每次取搜索方向的极值点，结合Johnson距离子算法，保证单纯形到原点的距离单调不增，在有限维空间中必然收敛。第三，判定的完备性：返回true时单纯形确实包围了原点；返回false时新支持点证明整个Minkowski差都在原点的某一侧，不可能包含原点。整个过程无需穷举所有顶点。

## 第14页 | GJK —— 具体示例演示（2D）

### 画面显示

- 标题：**GJK 实例演示 —— 两个2D凸多边形**
- 设两个正方形（动画分步展示）：

```
正方形A（中心在原点，边长2）：顶点 (-1,-1), (1,-1), (1,1), (-1,1)
正方形B（中心在(1.5, 0)，边长2）：顶点 (0.5,-1), (2.5,-1), (2.5,1), (0.5,1)

→ A和B在X轴方向上有0.5的重叠 → 应判定为相交
```

- **GJK执行过程（分步动画）**：

| 步骤 | 操作 | 单纯形状态 | 搜索方向 |
|------|------|-----------|---------|
| 1 | 取初始方向 d=(1,0)，Support = A中最右(1,0) − B中最左(0.5,−1) = **(0.5, 1)** | [(0.5,1)] | 方向改为 −(0.5,1) = (−0.5,−1) |
| 2 | Support沿(−0.5,−1)：A中最远 = (−1,−1)，B中最远 = (2.5,1)，差 = **(−3.5, −2)** | [(0.5,1), (−3.5,−2)] | 线段，计算新方向朝向原点 |
| 3 | 线段(0.5,1)→(−3.5,−2)与原点的关系 → 计算新搜索方向 | — | — |
| 4 | 取新支持点 → 形成三角形 → 检查原点是否在三角形内 | [(0.5,1), (−3.5,−2), **新点**] | **原点在三角形内 → 返回 true** |

- 结论：**A ∩ B ≠ ∅，判定为相交 ✓**
- 底部：整个过程只用了3次Support调用，没有遍历任何物体的全部顶点

### 讲稿（~45秒）

> 让我们用具体例子走一遍GJK。正方形A中心在原点，B中心在(1.5, 0)，X方向重叠0.5，应判定相交。GJK从初始方向(1,0)取支持点(0.5, 1)，再改方向指向原点取第二个点(-3.5, -2)。两点线段不包围原点，取第三个点形成三角形后，检查原点在三角形内部，判定相交。全过程只调用3次Support函数——每次只需对两个正方形各找一次极值——远比穷举所有顶点对高效。

## 第16页 | 三种算法综合对比

### 画面显示

- 标题：**总结对比 —— 先快再准，先筛后算**
- 大对比表：

| 维度 | 四叉树 | SAP | GJK |
|------|--------|-----|-----|
| **定位** | 粗筛（空间索引） | 粗筛（区间扫描） | 精算（几何判定） |
| **设计策略** | 分治 | 降维+时间局部性 | 迭代逼近 |
| **最好时间** | O(n log n) | **O(n)**（后续帧） | O(1)（增量模式1-2次迭代） |
| **平均时间** | O(n log n) | **O(n + m)**，m=重叠数 | O(n)（Support遍历） |

| **适用维度** | 2D | 2D/3D | 2D/3D |

### 讲稿（~35秒）

## 第17页 | 拓展讨论

### 画面显示

- 标题：**拓展讨论 —— 从GJK出发的延伸**
- 三个方向（卡片式布局）：

**方向一：EPA（Expanding Polytope Algorithm，van den Bergen提出）**
- GJK只能判定"是否相交"，不能给出"穿透多深"
- EPA在GJK判定相交后，利用GJK最终的单纯形作为起点
- 核心原理：Minkowski差上距原点最近的点 = 穿透方向；该点到原点的距离 = 穿透深度
- EPA不断扩展多面体（simplex点数不限于4），逐步逼近Minkowski差边界上的最近点
- 应用：物理响应——物体被推开多少、往哪个方向推

**方向二：连续碰撞检测（CCD）**
- 离散检测：每个时间步检测一次，高速物体可能"穿墙"
- CCD：沿运动轨迹做扫掠体检测，t∈[0,1]连续求解
- 典型场景：子弹、高速赛车

**方向三：非凸体处理**
- GJK只适用于凸体
- 实际游戏模型多为凹形 → 凸分解（Convex Decomposition）
- V-HACD等算法将凹网格分解为多个凸体，再分别用GJK检测

- 底部：**GJK是起点，不是终点——从判定到响应、从离散到连续、从凸体到凹体，每一步都是新的算法问题**

### 讲稿（~45秒）

> 从GJK出发有多个延伸方向。第一是EPA——GJK只能告诉我们"碰了还是没碰"，但物理引擎需要知道"碰了多深"，EPA在GJK基础上扩展单纯形来计算穿透深度。第二是连续碰撞检测CCD——物体运动很快时离散帧检测可能漏碰，CCD沿运动轨迹做连续扫掠。第三是非凸体处理——GJK只处理凸体，游戏模型多为凹形，需先凸分解再交给GJK。这三个方向说明GJK是起点而非终点。
