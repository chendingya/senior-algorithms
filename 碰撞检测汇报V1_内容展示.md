## 1. 问题从哪里来 —— 穿模问题和实时检测困境

- 玩家看到的是穿模、卡墙、空气墙，开发者背后面对的是碰撞检测。
- 如果场景里有一万个对象，暴力两两检测会把单帧复杂度推到 **O(N²)**。
- 真正难的不是两个物体能不能比，而是整张地图里谁值得进入下一轮。

> 游戏算法千千万，碰撞检测很难办

---

## 2. 从暴力法到碰撞流水线 —— 粗筛挂号，精算问诊

### 暴力互查的问题

- 每个对象都和其他所有对象做比较。
- 对象越多，冗余计算越夸张。
- CPU 花大量时间处理根本不可能相撞的组合。

### 粗筛阶段

- 用轻量结构快速剔除远距离对象。
- 输出的是候选碰撞对，不是最终答案。
- **四叉树**和 **SAP** 都属于这一层。

### 精算阶段

- 只对嫌疑对象做真正的几何判断。
- **GJK** 把双体相交问题改写成原点判定问题。
- 目标是更准，而不是更广。

> 粗筛缩范围，精算给结论。

## 3. 四叉树的直觉

- 四叉树把整张 2D 地图不断切小，让对象优先和局部邻居打交道。
- 当某个区域内对象超过阈值，节点就触发分裂，继续递归下沉。
- 查询时不再看全图，而是只看目标叶子节点与相交区域。

| 核心动作 | 适用场景 | 风险点 |
|---------|---------|-------|
| 插入后分裂 | 2D 偏静态 | 动态维护重 |

> 一尺之树，日分其四，万世不竭

---

## 4. 四叉树的取舍

> 我又回想起了，被树的维护折磨的那些天

### 优点

- 结构直观，适合二维地图和区域查询。
- 对象分布稀疏时，能显著缩小比较范围。

### 缺点

- 对象剧烈移动时，树会频繁删除、分裂、重组。
- 最坏情况下树深会膨胀，维护成本吞掉收益。

### 伪代码

```
func Insert(n *Node, obj Obj)
    if n.count < MaxCap
        n.objs = append(n.objs, obj); return
    if !n.isDivided { n.Subdivide() }
    for _, child := range n.children
        if child.Contains(obj) {
            Insert(child, obj); return

func Query(n *Node, rng Range, res *[]Obj)
    if !n.Intersects(rng) return
    for _, obj := range n.objs
        if rng.Contains(obj)
            *res = append(*res, obj)
    if n.isDivided
        for _, child := range n.children
            Query(child, rng, res)
```

## 5. 从四叉树到 SAP —— "降维打击"

- SAP 把每个对象的 AABB 投影到一根坐标轴上，先做一维重叠判断。
- 只要两个区间在轴上没有重叠，它们就不值得进入下一轮。
- 对高动态场景来说，这种做法往往比反复维护树结构更划算。

| 核心动作 | 适用场景 | 关键优势 |
|---------|---------|---------|
| 投影与扫描 | 高动态对象 | 复用排序 |

---

## 6. SAP 伪代码与帧间一致性

> 序定，则帧帧无虞

### 为什么能接近线性

- 上一帧排好的顺序，下一帧通常还差不多。
- 对象只是小幅移动时，插入排序几乎就是线性时间。

### 什么时候会吃亏

- 对象突然乱序、剧烈穿插时，排序成本会重新抬头。
- 只做轴向重叠判断，仍然需要精算阶段给最终答案。

### 伪代码

```
func SAP(objs List, res List)
    // 1. 沿某一轴（如X轴）按最小边界进行排序
    Sort(objs, key=minX)
    // 2. 遍历排序后的对象
    for i = 0 to len(objs) - 1
        curr = objs[i]
        // 3. 检查后续对象是否重叠
        for j = i + 1 to len(objs) - 1
            next = objs[j]
            // 4. 剪枝：若后续对象的最小X > 当前的最大X，直接结束内循环
            if next.minX > curr.maxX
                break
            // 5. 若包围盒确实相交，则记录碰撞对
            if Intersects(curr, next)
                res.append(curr, next)
```

## 7. 闵可夫斯基差 —— 两体相交，以差判之，决于原点

### 定义

$$A \ominus B = \{ a - b \mid a \in A,\ b \in B \}$$

- 把 A 的点减去 B 的点，会得到一团新的点云。
- 如果这团点云包住原点，就等价于 A 和 B 相交。
- 这样一来，后面的 GJK 就只需要围着原点做逼近。

---

## 8. GJK 的流程 —— 取支、立形、决原

- GJK 不会穷举全部边和点，而是每次只取一个最有价值的支持点。
- 这些支持点会逐步拼出线段、三角形等单纯形结构。
- 只要单纯形围住原点，就能判定两个凸体已经相交。

| 第一步 | 第二步 | 第三步 |
|-------|-------|-------|
| 找支持点 | 更新单纯形 | 检查原点 |

---

## 9. GJK 伪代码与判定逻辑 —— 只取极点，不看全体

### GJK 的要点

- GJK 适合凸体，相交判定快，工程上很常见。
- 如果物体是非凸的，通常先做凸分解，再交给 GJK。
- 当需要求穿透深度时，还可以继续接 EPA。

### 伪代码

```
func GJK(shapeA, shapeB)
    // 1. 获取初始支撑点，加入单纯形
    dir = InitialDirection()
    simplex = [Support(shapeA, shapeB, dir)]
    // 2. 搜索方向改为朝向原点
    dir = -simplex[0]
    // 3. 迭代寻找
    while true
        p = Support(shapeA, shapeB, dir)
        // 4. 剪枝：若新点未能跨越原点，则不可能相交
        if Dot(p, dir) <= 0  return false
        simplex.append(p)
        // 5. 更新单纯形并确定新方向，若包含原点则相交
        if UpdateSimplex(simplex, ref dir)  return true

func Support(shapeA, shapeB, dir)
    // 闵可夫斯基差：A沿dir的最远点 - B沿相反方向的最远点
    return shapeA.Farthest(dir) - shapeB.Farthest(-dir)
```

## 10. 工业界怎么组合使用

> 小孩子才做选择，成熟的引擎什么都要

### Unity

- 3D 物理常见于 PhysX 方案。
- 粗筛可配置为多种 broadphase 形式。

### Box2D

- 2D 物理常见实现。
- 广相阶段常依赖动态 AABB 树。

### UE (Unreal Engine)

- Chaos 提供空间加速结构。
- 精算侧可暴露 GJK 相关接口。

### 结论

- 四叉树更像逻辑层或自定义空间索引工具。
- SAP、动态树、空间加速结构才是现代引擎更常见的组合拳。
- 真正的工程设计，重点是按场景挑结构，而不是押宝单一算法。

## 11. 总结 —— 回本溯源

| 算法 | 定位 | 强项 | 短板 |
|-----|------|------|------|
| **四叉树** | 二维粗筛 | 局部查询直观 | 高动态场景维护重 |
| **SAP** | 动态粗筛 | 排序结果可复用 | 极端乱序时收益下降 |
| **GJK** | 凸体精算 | 只取极点也能高效判定 | 非凸体要先处理 |

> **一言以蔽之：先快再准；先筛后算。**
